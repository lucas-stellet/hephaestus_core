# ADR-002: Edge extraction — abordagem hibrida @before_compile + __on_definition__

## Status

Accepted

## Date

2026-04-05

## Context

O macro `use Hephaestus.Workflow` precisa extrair edges do grafo a partir das clauses de `transit/2` (estatico) e `transit/3` (dinamico com `@targets`). A decisao 13 do design doc estabelece que edges sao acumuladas via `Module.register_attribute` em vez de extraidas do AST arbitrario. Porem, o mecanismo exato de captura precisa ser definido: `__on_definition__/6` vs `@before_compile` + `Module.get_definition/2` vs override de `def` via macro.

## Decision

Abordagem hibrida:

- **transit/2**: extraido no `@before_compile` via `Module.get_definition(env.module, {:transit, 2})`. Neste ponto, aliases estao resolvidos pelo compilador. O body e literal por contrato (sem logica), entao a extracao do AST e segura.

- **transit/3**: capturado via `__on_definition__/6`. Quando o callback ve uma clause de transit/3, le `Module.get_attribute(env.module, :targets)`, valida que existe, e acumula em `:hephaestus_dynamic_edges`. O body pode ter logica livre — nao e parseado.

No `@before_compile`, merge das duas fontes para construir o DAG completo via `libgraph`.

## Alternatives Considered

### Alternative 1: Tudo via __on_definition__

- **Description**: Usar `__on_definition__/6` para ambos transit/2 e transit/3
- **Pros**: Mecanismo unico, sem duplicacao
- **Cons**: No `__on_definition__`, aliases nos args podem nao estar resolvidos. O body de transit/2 tambem pode ter aliases nao expandidos. Requer `Macro.expand/2` em multiplos pontos
- **Why rejected**: `Module.get_definition/2` no `@before_compile` garante aliases resolvidos para transit/2, que e o caso mais comum

### Alternative 2: Tudo via @before_compile + Module.get_definition

- **Description**: Extrair tudo no `@before_compile` usando `Module.get_definition` para ambas aridades
- **Pros**: Aliases resolvidos em todos os casos
- **Cons**: Para transit/3, nao ha como associar `@targets` a cada clause especifica no `@before_compile` — o attribute ja foi consumido ou sobrescrito
- **Why rejected**: `@targets` precisa ser capturado no momento da definicao da clause, o que requer `__on_definition__`

### Alternative 3: Override de def via macro

- **Description**: `use Hephaestus.Workflow` importa macro que substitui `def` e intercepta `def transit`
- **Pros**: Controle total sobre a captura
- **Cons**: Overridar `def` e invasivo, pode interferir com outros macros, e fragil para manutencao
- **Why rejected**: Muito invasivo. `__on_definition__` e o mecanismo oficial do Elixir para este caso

## Consequences

### Positive

- transit/2 tem aliases 100% resolvidos via `Module.get_definition/2`
- transit/3 captura `@targets` no momento exato da definicao — sem ambiguidade
- Cada mecanismo faz o que faz melhor

### Negative

- Dois mecanismos diferentes para o mesmo conceito (edge extraction) — mais complexidade no macro
- `__on_definition__` para transit/3 ainda precisa de `Macro.expand/2` nos pattern args

### Risks

- `Module.get_definition/2` retorna AST — se alguem escrever logica no body de transit/2 (violando o contrato), a extracao falha. Mitigacao: validar no `@before_compile` que o body e extraivel e dar erro claro
- `__on_definition__` recebe args como AST com aliases possivelmente nao expandidos. Mitigacao: `Macro.expand/2` com o env do modulo
