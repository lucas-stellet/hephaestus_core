# ADR-001: Arquitetura do Hephaestus Core — Functional Core + OTP Runtime com DAG

## Status

Accepted

## Date

2026-04-03

## Context

O Hephaestus v1 é uma aplicação Phoenix completa (UI, banco próprio, REST API, CLI) com workflows construídos em runtime via API. A decisão é transformá-lo numa lib Elixir pura de workflow engine, com definição em compile-time via callbacks, sem banco de dados, sem UI, e sem adapters forçados.

Precisávamos definir como o engine executa workflows e gerencia estado de instâncias, considerando suporte a steps assíncronos (wait, wait_for_event) e execução paralela (fan-out/fan-in em DAGs).

## Decision

Adotar uma arquitetura em camadas seguindo o padrão "Do Fun Things with Big Loud Worker-Bees":

- **Data**: Structs puras (Workflow, Instance, Step, Transition) com `active_steps` e `completed_steps` como MapSets para suportar DAGs
- **Functional Core**: Engine puro sem side-effects — executa steps, resolve transições, computa fan-in via `MapSet.subset?`. Testável em IEx sem nenhum processo OTP
- **Boundary**: GenServer por instância de workflow, wrapping o core com thin callbacks
- **Lifecycle**: DynamicSupervisor + Registry para gerenciar instâncias
- **Workers**: Task.Supervisor para execução paralela de branches em fan-out

Workflows definidos via callbacks (`steps/0`, `transitions/0`, `initial_step/0`) com validação de grafo em `@before_compile`. Transitions aceitam atom (sequencial) ou lista de atoms (fan-out). Fan-in resolvido por predecessores computados em compile-time.

Storage via behaviour com implementação ETS como default.

## Alternatives Considered

### Alternative 1: Engine Síncrono-first com Async Opt-in

- **Description**: Core funcional puro, sem processos. Steps async pausam e retomam via chamada explícita. Sem GenServer por instância.
- **Pros**: Máxima simplicidade, sem overhead de processos
- **Cons**: App precisa gerenciar toda a infraestrutura de retomada. Sem paralelismo real no core.
- **Why rejected**: Steps async são essenciais e a execução paralela (DAG) entra no MVP. Sem processos, a app assumiria complexidade demais.

### Alternative 2: Process-per-Instance sem Core Funcional

- **Description**: Cada instância é um GenServer com lógica de engine embutida nos callbacks.
- **Pros**: Modelo mental simples — workflow = processo
- **Cons**: Lógica de negócio misturada com gestão de processos. Difícil de testar. Viola a separação de camadas.
- **Why rejected**: Testabilidade e separação de concerns são prioridade. O core deve funcionar sem OTP.

### Alternative 3: Puramente Funcional sem Processos

- **Description**: Engine é função pura que retorna estado. Zero side-effects, zero processos. App gerencia tudo.
- **Pros**: Máxima testabilidade e composabilidade
- **Cons**: Sem suporte real a async ou paralelismo — a app precisa construir toda a infraestrutura
- **Why rejected**: Coloca peso demais no consumidor da lib para funcionalidades que são core do workflow engine.

## Consequences

### Positive

- Core testável em IEx sem supervision tree
- Separação clara: lógica pura no Engine, processos no Runtime
- DAGs com fan-out/fan-in suportados nativamente
- Validação de grafo em compile-time (ciclos, steps órfãos, joins impossíveis)
- Paralelismo real via Task.Supervisor no runtime

### Negative

- Mais código que um engine puramente funcional
- Dois "modos" de usar (core puro vs runtime) podem confundir inicialmente
- Fan-in adiciona complexidade ao estado da instância (MapSets)

### Risks

- Merge de context em fan-in pode ter conflitos (dois steps escrevem a mesma key) — mitigação: last-write-wins ou documentar que steps paralelos devem usar keys distintas
- GenServer por instância pode ser problema em escala muito alta — mitigação: PartitionSupervisor como evolução futura

## Implementation Notes

- Built-in steps: wait, wait_for_event, end, debug
- Condition step removido — qualquer step pode branchar retornando eventos diferentes
- Storage behaviour com implementação ETS no MVP
- Sem telemetry no MVP, sem banco de dados
- `use Hephaestus.Workflow` injeta `@before_compile` para validação de grafo e computação de `__predecessors__/1`
