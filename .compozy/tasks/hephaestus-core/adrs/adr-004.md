# ADR-004: Context como struct com namespace por step

## Status

Accepted

## Date

2026-04-03

## Context

O engine precisa propagar dados entre steps. Em workflows com fan-out/fan-in, steps paralelos podem escrever context_updates simultaneamente, criando risco de conflito de keys. Além disso, o context misturava dados iniciais da app com resultados de steps.

## Decision

Context é uma struct `Hephaestus.Core.Context` com dois campos:
- `initial` — dados passados pela app no `start_instance`, imutável durante a execução
- `steps` — map de resultados de steps, namespaced por step ref

Todo step (paralelo ou sequencial) escreve sob `context.steps.<ref>`. O engine usa `Context.put_step_result/3` para manter consistência.

## Alternatives Considered

### Alternative 1: Map solto com merge no root

- **Description**: `Map.merge(context, context_updates)` direto no root
- **Pros**: Simples, ergonômico para workflows lineares
- **Cons**: Conflito de keys em fan-in. Sem distinção entre dados iniciais e resultados de steps
- **Why rejected**: Conflitos em fan-in são silenciosos e difíceis de debugar

### Alternative 2: Namespace só para steps paralelos

- **Description**: Steps sequenciais fazem merge no root, paralelos escrevem sob `steps.<ref>`
- **Pros**: Mais ergonômico para workflows simples
- **Cons**: Duas regras diferentes. Dev precisa saber se o step vai rodar em paralelo ou não
- **Why rejected**: Inconsistência — mover um step de sequencial pra paralelo mudaria como o context é acessado

## Consequences

### Positive

- Zero conflito possível em fan-in
- Separação clara entre dados da app e resultados de steps
- Consistência — mesma regra para todo step
- Steps sabem exatamente de onde vem cada dado: `context.initial.x` ou `context.steps.step_ref.y`

### Negative

- Mais verboso que `context.x` direto — sempre precisa de `context.steps.step_ref.x` ou `context.initial.x`

### Risks

- Nenhum risco significativo identificado

## Implementation Notes

- `Context.new(initial_data)` cria o context
- `Context.put_step_result(context, step_ref, result)` adiciona resultado de step
- `context.initial` é read-only após criação
- Step `Debug` mostra ambos (`initial` e `steps`)
