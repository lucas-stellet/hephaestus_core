# ADR-003: Runner como adapter para execução de workflows

## Status

Accepted

## Date

2026-04-03

## Context

O Hephaestus precisa executar workflows com steps síncronos e assíncronos (wait, wait_for_event) e com paralelismo (fan-out). No MVP, a execução é via GenServer + DynamicSupervisor. No futuro, deve ser possível trocar para Oban (jobs com persistência, retry, multi-node) sem alterar o engine.

## Decision

Criar um Runner behaviour como adapter de execução. O runner encapsula toda a infraestrutura de execução: como instâncias são startadas, como steps rodam, como pausa/resume funciona, e como timers são agendados.

Callbacks: `start_instance/3`, `resume/2`, `schedule_resume/3`.

MVP: `Runner.Local` — um único módulo que implementa o Runner behaviour + GenServer. Cada instância é um processo via DynamicSupervisor + Registry, com Task.Supervisor para fan-out paralelo.

Futuro: `Runner.Oban` — stateless, cada execução é um job. Estado vive no storage (Ecto). Timers são jobs agendados com `scheduled_in`.

Configurado no módulo de entrada: `use Hephaestus, runner: Hephaestus.Runner.Local`.

## Alternatives Considered

### Alternative 1: Scheduler adapter (só timers)

- **Description**: Adapter apenas para agendamento de timers (wait steps). Execução sempre via GenServer
- **Pros**: Escopo menor, mais simples
- **Cons**: Não permite trocar o modelo de execução inteiro. Com Oban, não é só o timer que muda — é tudo
- **Why rejected**: O ganho do Oban é execução inteira dentro de jobs (persistência, retry, multi-node), não só timers

### Alternative 2: Sem adapter (GenServer fixo)

- **Description**: Execução hardcoded via GenServer. Futura migração pra Oban requereria reescrita
- **Pros**: Menos abstrações no MVP
- **Cons**: Lock-in no modelo de execução. Migração futura seria breaking change
- **Why rejected**: O custo do behaviour é mínimo e evita reescrita futura

## Consequences

### Positive

- Trocar de GenServer pra Oban é mudar uma linha de config
- Engine funcional fica 100% desacoplado de como é executado
- Runner.Local serve como referência pra implementações futuras

### Negative

- Runner.Local acumula responsabilidades (behaviour + GenServer num módulo)

### Risks

- O behaviour pode precisar de callbacks adicionais quando Oban for implementado — mitigação: manter o behaviour mínimo e estender quando necessário

## Implementation Notes

- Runner.Local é um módulo que implementa `@behaviour Hephaestus.Runtime.Runner` e `use GenServer`
- Registra instâncias via `{:via, Registry, {Hephaestus.Registry, instance_id}}`
- Fan-out via `Task.Supervisor.async_nolink/3`
- Futuro Runner.Oban seria módulo puro (sem GenServer), inserindo jobs via `Oban.insert/1`
