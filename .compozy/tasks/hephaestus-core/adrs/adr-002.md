# ADR-002: Protocolo StepDefinition para extensibilidade de tipos de step

## Status

Accepted

## Date

2026-04-03

## Context

O Hephaestus precisa suportar diferentes grupos de steps (genérico, channel, connector, custom) onde cada grupo tem seus próprios campos obrigatórios e configuração. O engine precisa interagir com qualquer tipo de step de forma uniforme, sem conhecer os tipos concretos.

## Decision

Usar o protocolo Elixir `Hephaestus.StepDefinition` como contrato entre o engine e qualquer tipo de step. Cada grupo de step define sua própria struct com `@enforce_keys` e implementa o protocolo. O engine interage exclusivamente via protocolo, nunca com structs concretas.

O protocolo define: `ref/1`, `module/1`, `config/1`, `transitions/1`.

Steps no workflow são declarados como lista de structs que implementam o protocolo, dentro de um callback único `definition/0` que retorna `%Hephaestus.Core.Workflow{}`.

## Alternatives Considered

### Alternative 1: Callbacks separados (steps/0, transitions/0, initial_step/0)

- **Description**: Três callbacks retornando maps simples
- **Pros**: API simples, menos abstrações
- **Cons**: Steps e transitions desacoplados. Sem tipagem dos grupos de step. Map solto sem validação de campos
- **Why rejected**: Não garante campos obrigatórios por grupo de step. Transitions separados dos steps dificulta leitura do fluxo

### Alternative 2: Behaviour com config struct por step

- **Description**: Cada step module declara `config_schema/0` retornando o módulo da struct de config
- **Pros**: Validação de config em compile-time
- **Cons**: Não resolve a questão de grupos/tipos de steps. Config é só um aspecto do step
- **Why rejected**: Protocolo é mais abrangente — cobre config, ref, module, transitions e é extensível pra qualquer campo futuro

## Consequences

### Positive

- Qualquer lib (hephaestus_channels, hephaestus_connectors) pode criar seus próprios tipos de step com campos específicos
- Engine desacoplado de tipos concretos
- @enforce_keys por grupo garante campos obrigatórios em compile-time
- Extensível sem modificar o core

### Negative

- Mais indireção que maps simples
- Cada grupo de step precisa implementar o protocolo (boilerplate)

### Risks

- Protocolo dispatch tem custo (mínimo) vs acesso direto a struct — irrelevante pra workflow engine

## Implementation Notes

- `Hephaestus.Core.Step` é o step genérico do core com implementação padrão do protocolo
- `Hephaestus.Steps.ChannelStep` virá com `hephaestus_channels`
- Steps no workflow são lista, não map — o engine converte pra map indexado por ref no @before_compile
